# 2026-05-12 技术日记

## 今日主题：Claude Code 团队工程师：我为什么放弃 Markdown，全面转向 HTML

### 背景

今天在掘金社区看到一篇热门文章，讲的是 Claude Code 团队的一位工程师分享他们为何将文档和内部内容从 Markdown 迁移到 HTML。这个话题引发了不少讨论，也让我重新思考了文档格式的选择。

### 核心观点

#### 1. Markdown 的局限性

Markdown 设计初衷是简洁的纯文本标记，但在实际工程中暴露出一些问题：

- **语义表达有限**：Markdown 缺乏丰富的语义标签，难以表达复杂的文档结构（如 `<aside>`、`<details>`、`<figure>` 等）
- **扩展不统一**：不同 Markdown 解析器对 GFM、MDX 等扩展的实现不一致，容易出现兼容性问题
- **嵌入交互困难**：在 Markdown 中嵌入交互式组件（折叠面板、Tab 切换等）需要依赖额外插件或 HTML 混写

#### 2. HTML 的优势

- **语义化标签**：原生支持丰富的语义化元素，有助于可访问性和 SEO
- **组件化友好**：与 React/Vue 等框架的 JSX/模板天然契合
- **精细控制**：可以直接控制样式和交互，不依赖解析器的二次转换
- **工具链成熟**：有大量成熟的工具链支持（如 DOM 解析、AST 操作）

#### 3. 实际权衡

当然这不是说 Markdown 一无是处：

- **快速写作**：对于简单的 README、笔记，Markdown 仍然是最高效的选择
- **可读性**：纯文本格式在未渲染状态下也非常好读
- **团队协作**：Markdown 的学习成本低，适合非技术人员参与

关键在于场景选择——内部工具文档、需要复杂交互的文档适合 HTML；而简单说明、代码注释、快速笔记则 Markdown 更合适。

### 个人思考

这个迁移决策背后的逻辑很务实：

1. **工具服务于工作流**：不要因为"习惯"而拒绝更好的工具
2. **渐进式迁移**：不是一夜之间全部重写，而是新内容用新格式，旧内容逐步迁移
3. **文档即代码**：HTML 可以更好地融入 CI/CD 流程，做自动化检查和部署

这也让我反思自己项目中的文档管理方式。目前我在 daily-log 项目中用 Markdown 写日记，对于简单的日记格式来说完全够用。但如果未来要加入更多交互元素（比如可折叠的代码块、动态图表），可能需要考虑 MDX 或直接用 HTML。
