## Spring Boot 事务传播行为小记

今天在排查一个业务异常时，发现事务传播行为配置不当导致数据不一致的问题，特此记录。

### 问题场景

```java
@Service
public class OrderService {
    @Autowired
    private PaymentService paymentService;
    
    @Transactional
    public void createOrder(Order order) {
        // 保存订单
        orderRepository.save(order);
        // 调用支付服务
        paymentService.processPayment(order);
    }
}

@Service
public class PaymentService {
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void processPayment(Order order) {
        // 支付逻辑
        paymentRepository.save(payment);
        // 模拟异常
        if (payment.getAmount() <= 0) {
            throw new InvalidAmountException("金额必须大于0");
        }
    }
}
```

### 问题分析

当`processPayment`抛出异常时，由于使用了`REQUIRES_NEW`，支付事务会回滚，但订单事务却提交了。这导致订单创建成功但支付失败的不一致状态。

### 解决方案

1. **统一事务边界**：将关键业务放在同一个事务中
2. **异常处理**：在调用方捕获异常并决定是否回滚
3. **补偿机制**：引入事务补偿或消息队列最终一致性

### 经验总结

- `REQUIRES_NEW`会挂起当前事务，创建新事务，适合日志记录等辅助操作
- 核心业务流程建议使用默认的`REQUIRED`，确保在同一事务中
- 跨服务调用要考虑分布式事务方案（如Seata、MQ最终一致性）

---

*今日收获：事务传播行为的选择要结合业务场景，不能盲目使用REQUIRES_NEW。*
